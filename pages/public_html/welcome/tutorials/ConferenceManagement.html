A Conference Management Online Software

A lot of conferences use a Content Management System (CMS) to implement their Web site. And they use EasyChair (for instance) to manage papers. And this is a good solution ! However, such a solution requires a lot of manual work. For instance, one has to establish a one-to-one correspondance between registrations and papers publications. EasyChair (for instance) does not manage registrations. And this is only one example showing that many Excel files are still required to manage the conference.

What we propose here is a fully integarated Conference Management Online Software that will be refrerred as CMOS. CMOS will provide all required functions:

  • Users lifecycle: account creation, management, etc.

  • Committees, dates,  etc.

  • Papers lifecycle: submissions, assignments, reviews, acceptance, etc.

  • Registrations and payments.

  • Statistics, reporting (Excel and PDF files), mailing, etc.

Our goal will be reached when nearly all operations will be done online.

Our model. We take as a model the wellkown Web Technologies & Internet Application (Webtech) confernce. We redesigned the screen with a 3-clolms design:

Page organisation. On this page, we see several frames:

  • The Header frame at the top of the page. It will be always displayed the same way.

  • The LoginLogout frame on the left of the page with the two red buttons. Its content will change depending the user is logged or not.

  • The Navigation frame on the left of the page under the Login frame. Its content will never change.

  • The Announces frame on the right of the page. It will be always displayed the same way.

  • The Footer frame at the bottom of the page. It will be always displayed the same way.

  • The Main frame at the center of the page. Its content depends on the page the user is looking at.

The Main frame is a DIV in XHTML meanwhile the other frames are IFRAME. The technical reason for that is not related to SAW-SDK but to usual Web programming techniques. It is that IFRAMEs have a src attributes. If the IFRAME is loaded once with the browser on one page and the same IFRAME appears in another page, when loading the other page, the content is already cached on the client side and will not require a HTTP transaction. Following this methods, the HTML code of all pages will look as follows: 

<html>
  <head>
    <title>WebTech 2012</title>
    <link rel='stylesheet' type='text/css' media='screen' href='../styles/CSS.class'/>
    <link rel="shortcut icon" type="image/png" href="../favicon.png">
  </head>
  <body class='main'>
    <div id='container'>
      <iframe id='header'      allowtransparency='true' src='../page/Header.class'></iframe>
      <iframe id='announces'   allowtransparency='true' src='../page/Announces.html'></iframe>
      <iframe id='loginlogout' allowtransparency='true' src='../page/LoginLogout.class'></iframe>
      <iframe id='navigation'  allowtransparency='true' src='../page/Navigation.html'></iframe>
      <div id='main'>

	<!-- Main frame content -->

      </div>
      <iframe id='footer'      allowtransparency='true' src='../page/Footer.html'></iframe>
    </div>
  </body>
</html>

In this tutorial, we will not explain CSS and JavaScript techniques for positionning, sizing and colouring the frames. To learn that, Google or another is your friend.

Code organization. All the Session Binz of the application will be under SAW-SDK/pages/public_html/conference. More precisely, they will be distributed in subdirectories of this directory. All the core classes (Entity Binz, Dynamic Elements, etc.) required by the application will be in the packages org.site.conference.
